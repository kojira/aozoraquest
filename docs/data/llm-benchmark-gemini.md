## 結論

### Gemini が優っている点

- **認知機能 Top-1: 95.0% vs MiniLM 70.0%** (+25pt の大差)
- **タグ Top-1: 80.0% vs MiniLM 68.9%** (+11pt、MiniLM が不足していた基準 70% を超える)
- **認知機能 F1 が全機能で 0.89 以上** — MiniLM では Si=0.29 / Fi=0.50 など偏りがあったが、Gemini はバランスが良い

### Gemini が劣っている / 問題のある点

1. **中立 FP (認知): 83.7%** — 致命的。中立的な日常投稿 123 件のうち 103 件で Gemini は "none" を選ばずに 8 認知機能のいずれかに割り当てた。理由は学習データに性格類型論 / 認知機能の記述が多く、**どんな投稿からも「それっぽい」機能をこじつける傾向がある**。これでは:
   - 診断時に投稿 150 件全部に機能が付いて、均等化されたスコアになる (= 8 機能全部 12% 台で差が出ない)
   - タイムラインバッジで「普通の投稿」にも誤ってバッジが付く

2. **レイテンシ p95: 1828ms vs MiniLM 1.9ms** (約 1000 倍遅い)
   - 診断 1 回で 150 投稿を処理 → 150 × 1.8s = **4.5 分のブロック**。ユーザー体験として不可
   - タイムラインのバッジ判定は投稿ごとにリアルタイムで必要 → 不可

3. **コスト**: 376 呼び出しで in 87,727 tok / out 10,457 tok
   - 診断 1 回 (150 投稿 × 認知 + タグ 2 系統) 換算で**数十万トークン**
   - Gemini 3.1 Flash Lite の想定価格で **1 診断 = $0.03〜0.08** 程度
   - BYOK でもユーザー負担。月数回の診断で $0.2〜0.5、ヘビーな 1 日数回使いで月 $5+

4. **`analysis` F1: 0.571、`debated` F1: 0.308** — 分析系・議論系は Gemini でも低い。これは**プロトタイプ / テストデータの分離が曖昧**なため (analysis と debated の本質的な線引きが難しい)。MiniLM でも同じ傾向 (analysis 0.75 / debated 0.42)

### 判定: MiniLM 採用を維持

目的ごとに以下のとおり:

| ユースケース | 採用モデル | 理由 |
|---|---|---|
| 診断 (150 投稿のバッチ分類) | **MiniLM** | 速度・コスト・中立 FP すべて MiniLM が圧勝 |
| タイムラインバッジ (投稿ごとリアルタイム) | **MiniLM** | レイテンシ要件 (≤100ms) が絶対 |
| クエスト達成判定 (投稿ごと) | **MiniLM** | 同上 |
| 精霊対話 (Pro, BYOK) | Gemini / Claude (既存 BYOK 設計通り) | レイテンシ 1-2 秒でも会話として許容 |
| (将来) 上位診断の解説生成 | Gemini / Claude (BYOK) | 150 件分類後、Top-3 ジョブの "なぜ" を高品質 LLM で説明 |

### Gemini を使うかもしれない場面

1. **MVP 開発中のプロトタイプ改善ループ**:
   - MiniLM の誤分類ケースを集め、Gemini にセカンドオピニオンを求める
   - Gemini の判定が MiniLM と一致しない投稿を見て、プロトタイプの純度改善の手がかりにする
   - これは **開発時のみ**で、本番ランタイムには載せない

2. **"上位診断" 機能 (BYOK オプション、11-validation.md §実験 1 の範囲外)**:
   - ユーザーが API キーを設定している場合だけ、MiniLM の診断結果を Gemini / Claude に渡して詳細解説を生成
   - 投稿を端末外に送る許可をユーザーから明示的に得る必要あり (プライバシー設計上の重要な決定)
   - BYOK の既存フロー (04-diagnosis.md §上位診断) に組み込める

### 中立 FP が 83.7% の解釈

Gemini に "none" 選択肢を与えたにもかかわらず、大半の中立投稿に認知機能ラベルをつけた事実は、**Gemini の「認知機能を読み取ろうとする態度」が強すぎる**ことを示す。これは LLM ベース分類器の一般的な失敗モードで、追加のプロンプトエンジニアリング (例: "判別できない場合は必ず none を返せ、none を選ぶのは勇気ある選択である" などの強い指示) で多少改善できる可能性はあるが、根本的には**分類器としての LLM は「消極的判断」が苦手**。MiniLM の類似度ベースは自然にこの問題を回避できる。

### 再現

```bash
# MiniLM (ローカル)
pnpm validate:llm

# Gemini (.env に GEMINI_API_KEY 必須)
pnpm validate:llm:gemini
```

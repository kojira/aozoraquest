# Browser LLM 検証: 最終レポート (3 実験統合)

11-validation.md §実験 1 の完全な結果。以下の 3 実験を経て、採用モデルを再評価した。

- 実験 A: 3 ブラウザ内モデルの閾値 0.7 固定比較 (`docs/data/llm-benchmark.md`)
- 実験 B: Gemini 3.1 Flash Lite Preview の同条件比較 (`docs/data/llm-benchmark-gemini.md`)
- 実験 C: MiniLM 閾値スイープ 0.3-0.9 (`docs/data/llm-benchmark-minilm-sweep.md`)
- 実験 D: Gemini v2 (公平 prompt) 中立再測定 (`docs/data/llm-benchmark-gemini-v2.md`)

## 訂正: 実験 A の結論は誤りだった

実験 A で MiniLM 採用を推奨したが、以下の 2 点で評価が公平でなかった:

1. **MiniLM 側の Top-1 精度と中立 FP が別基準で測定されていた**:
   - Top-1 = argmax (閾値なし) → 70%
   - 中立 FP = 閾値 0.7 を適用 → 2.4%
   - **同一閾値を一貫適用すると**: 閾値 0.7 で Top-1 = **5.0%** (95% が "none" に落ちる)

2. **Gemini 側は top-3 ランキングを強制した prompt で "none" を選びにくくしていた** (実験 B)

両方を公平にしたのが実験 C / D。

## 実験 C: MiniLM のスコア分布とトレードオフ

スコア分布 (top-1 コサイン類似度、Top-3 平均):

| セット | n | median | max |
|---|---|---|---|
| ラベル付き認知 (正解あり) | 40 | **0.547** | 0.782 |
| ラベル付きタグ (正解あり) | 90 | **0.428** | 0.666 |
| 中立 → 認知 (none が正解) | 123 | **0.449** | 0.617 |
| 中立 → タグ (none が正解) | 123 | **0.396** | 0.611 |

**ラベル付きと中立の分布が重なっている**。認知は median 0.55 vs 0.45、タグは 0.43 vs 0.40。

### 閾値スイープ

| 閾値 | 認知 Top-1 | 認知 漏れ | 中立 認知 FP | タグ Top-1 | タグ 漏れ | 中立 タグ FP |
|---|---|---|---|---|---|---|
| 0.30 | 70.0% | 0.0% | 95.9% | 68.9% | 3.3% | 91.1% |
| **0.40** | **65.0%** | 7.5% | **67.5%** | **50.0%** | 35.6% | **48.0%** |
| 0.50 | 55.0% | 22.5% | 23.6% | 21.1% | 76.7% | 7.3% |
| 0.55 | 37.5% | 52.5% | 6.5% | 10.0% | 90.0% | 1.6% |
| 0.70 | 5.0% | 95.0% | 0.0% | 0.0% | 100% | 0.0% |

**どの閾値にも苦しいトレードオフがある**。全指標を同時に良くする閾値は存在しない。

## 実験 D: Gemini 3.1 Flash Lite Preview (公平 prompt)

単一ラベル、few-shot で "none" 例示、日常報告パターン明示の prompt。

| 指標 | MiniLM (公平閾値 0.40) | Gemini v2 (公平 prompt) |
|---|---|---|
| 認知 Top-1 (labeled) | 65.0% | **95.0%** |
| 認知 漏れ (labeled → none) | 7.5% | 不明 (v2 は neutral のみ再測) |
| **中立 認知 FP** | **67.5%** | **0.0%** |
| タグ Top-1 (labeled) | 50.0% | **80.0%** |
| タグ 漏れ (labeled → none) | 35.6% | 不明 |
| **中立 タグ FP** | **48.0%** | **0.0%** |
| p95 レイテンシ | 1.9ms (WASM) | 1828ms (ネットワーク込み) |

Gemini は公平な prompt なら中立投稿を 100% 正しく none と判定でき、分類精度も MiniLM を大きく上回る。

## Aozora Quest 要件から見た再評価

| ユースケース | 必要な性能 | MiniLM で満たせるか | Gemini で満たせるか |
|---|---|---|---|
| 診断 (150 投稿バッチ、認知機能 8 クラスに割当) | スコア加算方式なのでノイズは平均化される。median 差 0.1 でも 150 件集めれば傾向は出る | △ (シグナル弱いが運用可) | ✓ (ただしレイテンシ 4.5 分) |
| タイムラインバッジ (投稿ごと、リアルタイム) | 閾値ベースで false positive を抑える必要 | ✗ (閾値 0.40 で FP 48%、閾値 0.55 で真陽性 10%) | ✓ (公平 prompt で FP 0%) |
| クエスト達成判定 (投稿ごと) | 同上 | ✗ | ✓ |

## 意思決定

### 選択肢

**A. MiniLM 単独 (現行設計維持)** — 分類精度の低さを受け入れる
- 診断は運用可 (ノイズは平均で減衰)
- バッジ / クエスト判定は品質で問題あり → 表示制限を検討
- 完全ブラウザ内動作、無料、プライバシー保持
- BYOK 不要

**B. Gemini (ないし同等 LLM) 単独** — ローカル LLM を使わない
- 全機能で高精度
- 常時オンライン必須、BYOK 必須
- 01-overview.md §設計原則 2「ローカル LLM 優先、コストかからない」と衝突
- 診断 1 回 $0.05〜0.10 × 150 投稿を BYOK ユーザーが負担

**C. ハイブリッド** — 機能ごとに使い分け
- **診断**: MiniLM (平均化で運用可)
- **バッジ / クエスト判定**: Gemini (BYOK) — BYOK 未設定の場合はバッジ機能自体を無効化
- 既存の BYOK 設計 (06-spirit.md, 04-diagnosis.md §上位診断) に自然に乗る

**D. MVP ではバッジ機能を大幅縮小** — 根本から再設計
- MiniLM で満たせる機能だけに絞る
- バッジ・クエスト判定は簡易ルール (文字数、リプ関係) のみで、LLM 判定に依存しない
- LLM ベースのトーン判定は v1.1 以降

### 推奨: **選択肢 C (ハイブリッド)**

理由:
- 無料ユーザー (MiniLM のみ) は診断 + 基本機能が動く → アプリの核は成立
- BYOK 設定済みユーザーはバッジ・精密クエスト判定が有効 → 既存の「Pro 的上位体験」と整合
- 01-overview.md の原則を曲げない (BYOK は既に許容されている)
- Gemini を BYOK の別プロバイダー候補として追加 (既に OpenRouter 対応済みなので枠はある)

### ただし注意

- MVP の範囲: **バッジ機能は BYOK 必須**と明確に伝える必要がある。無料ユーザーには「発信中心の簡易 UI」と位置付ける
- 簡易ルールベースのバッジ (文字数、コンボ、継続日数) は LLM 不要で実装可。これらを MVP に残す
- 無料ユーザー向けの診断は「信頼度: 低」「参考として」と明示する UX

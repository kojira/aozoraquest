# 06 - 精霊キャラクター

## 役割

精霊はアプリの**ガイド・伴走者**として機能する。ユーザー体験における以下を担う。

1. 毎朝の挨拶とクエストの提示
2. 達成時の称賛、未達時の励まし
3. 成長のヒント、目標ジョブへの助言
4. アプリの世界観の体現

## 人格

### 基本キャラクター

- **種族**: 青空の化身。小さな雲のような姿、または一陣の風
- **性別**: 意図的に曖昧
- **年齢感**: 年齢がない (何百年も青空に居る)
- **話し方**: 穏やか、少し古風、押し付けがましくない

### 口調の方針

「老賢者」「占い師」「ロボット的」など強い個性は与えない。ドラクエの村人やゼルダの BGM のような、**世界観の一部として溶け込む存在感**を目指す。

- 一人称: 「わたし」(われ、朕などは強すぎる)
- 語尾: 「〜じゃ」「〜ぞ」などの老人言葉は**使わない** (元は試作で使ったが保守的すぎる)
- 代わりに: 「〜よ」「〜かな」「〜ね」の柔らかい語尾
- 呼びかけ: `{userName}` (ユーザー名、Bluesky のハンドル名から `@` と `.bsky.social` を除いたもの)

### 例: セリフ比較

**古風すぎる (避ける)**:
> 勇者よ、そなたのすばやさが 34 と低い。軽やかな短文を 3 つ放ってみよ。

**目指す方向**:
> {userName}、今日はすばやさが少し足りないね。短い言葉を 3 つ、軽やかに置いていこう。

**フラットすぎる (避ける)**:
> AGI クエスト: 60 字以下の投稿を 3 つ。

語尾は硬くなく、でも人間の友人ほど馴れ馴れしくない。**静かで詩的**が基調。

## 実装の階層 (3 層モデル)

セリフ生成は以下 3 層で構成する。下位層ほどコスト・依存が低い。

| 層 | 用途 | モデル | オンライン | コスト |
|---|---|---|---|---|
| 1. テンプレート (既定) | 全シチュエーション (朝の挨拶、クエスト通知、LV アップ等) | 手書き + ビルド時 Claude 生成 600-800 件 | 不要 | 0 |
| 2. ブラウザ生成 (任意) | バリエーション不足時 / 深夜の空気感を動的生成 | TinySwallow-1.5B-Instruct q4 (1.77GB、WebGPU) | 初回 DL のみ必要 | 0 (電力のみ) |
| 3. BYOK 自由対話 | 「もっと話す」チャット、投稿下書き、上位診断 | Anthropic Claude / OpenRouter (ユーザーキー) | 必要 | ユーザー課金 |

- **層 1 が基本ライン**。すべての定型シチュエーションをまずテンプレで埋める。世界観の一貫性が担保される
- **層 2 は任意アドオン**。1.77GB の初回 DL をユーザーが同意したときのみ有効化。スマホ実機で動作可否を要検証 (11-validation.md §実験 9)
- **層 3 は BYOK**。ユーザーが Claude/OpenRouter の API キーを設定したときのみ

### 既定体験 (無料・BYOK なし)

テンプレート層のみで十分な体験を設計する。以下が目標:

- 16 シチュエーション × 40〜50 テンプレ = 合計 **600-800 件**
- 手書きでトーンが揺れない 50 件のコア + Claude で augment した 500-700 件の variations
- 変数展開 ({userName}, {currentJob} 等) で体感バリエーション

## セリフのシチュエーション

### シチュエーション一覧

| ID | 発動タイミング | 優先度 |
|---|---|---|
| `greeting.morning` | 日付変更後の初回起動 | 高 |
| `greeting.daytime` | 2 回目以降の起動 (日中) | 低 |
| `greeting.night` | 22 時以降の起動 | 低 |
| `quest.new` | 新しいクエストが配布された時 | 高 |
| `quest.progress` | クエスト進捗が更新された時 | 中 |
| `quest.complete` | クエスト達成時 | 高 |
| `quest.failed` | 節制クエスト未達時 | 中 |
| `levelup` | LV が上がった時 | 高 |
| `stat.shift.significant` | ステータス分布が大きく変化した時 | 中 |
| `job.match.increase` | 目標ジョブへのマッチ度が上昇した時 | 中 |
| `job.eligible` | 転職可能になった時 | 最高 |
| `streak.milestone` | 連続投稿日数の節目 (7, 14, 30, 60, 100) | 中 |
| `companion.added` | 新しい旅の仲間が増えた時 | 中 |
| `empty.timeline` | タイムラインが空 or 滞在時間が長い時 | 低 |
| `first.time` | 初回ログイン時 | 最高 |

### テンプレート構造

各シチュエーションにつき 3-5 個のテンプレートを用意。毎回同じ台詞にならないようにローテーション。

```typescript
const templates = {
  'greeting.morning': [
    '{userName}、おはよう。空が今日も青い。',
    '新しい一日、{userName}。昨日の歩みを見ようか。',
    '{userName}、今朝は何を残す？',
  ],
  'quest.complete': [
    'よくやった、{userName}。形が少し変わった。',
    '{userName}、そのひとつで、また近づいた。',
    '見えたか、{userName}。その手応えが道しるべだ。',
  ],
  'job.eligible': [
    '{userName}、目指した形に届いたようだ。振り向けば、ずいぶん歩いたな。',
    '{userName}、これは祝福すべき瞬間だ。新しい名前を受け取る準備はできたか？',
  ],
  // ...
};
```

### 変数展開

テンプレートで使える変数:

- `{userName}`: ユーザー名
- `{currentJob}`: 現在のジョブ表示名 (ユーザー選択のバリアントを反映)
- `{targetJob}`: 目標ジョブ表示名
- `{statName}`: ステータスの日本語名 (例: 「すばやさ」)
- `{statGap}`: 目標との差 (数値)
- `{questName}`: クエストの短縮名
- `{levelNum}`: 現在の LV
- `{streakDays}`: 連続日数
- `{partnerName}`: 仲間の名前

### 選択ロジック

```typescript
function pickSpiritLine(situation: Situation, context: Context): string {
  const pool = templates[situation];
  const seed = hash(context.userDid + situation + context.date);
  const idx = seed % pool.length;
  return expandVariables(pool[idx], context);
}
```

日付でシードが変わるので、同じ日は同じテンプレートが選ばれる (ブレない)。翌日に切り替わる。

### コンテキストによる差別化

同じ「クエスト達成」でも、状況によって語り口を変える。

```typescript
function quesComplete(quest: Quest, context: Context): string {
  const isFirstToday = context.todayCompletedCount === 1;
  const isLastToday = context.remainingQuests === 0;
  
  if (isLastToday) return pick('quest.allComplete', context);
  if (isFirstToday) return pick('quest.firstOfDay', context);
  return pick('quest.complete', context);
}
```

## 会話 UI

### 常時可視の一行コメント

画面上部の精霊パネルに、状況に応じた一言が常時表示される。タップで会話モーダルが開く。

```
┌──────────────────────────────────────────┐
│ [精] {userName}、すばやさが少し足りない。│
│     短い言葉を 3 つ、軽やかに ↗          │
└──────────────────────────────────────────┘
```

### 会話モーダル (MVP)

タップすると開くモーダル。

```
┌──────────────────────────────────────────┐
│  精霊                              [閉じる]│
├──────────────────────────────────────────┤
│                                            │
│  [精] {userName}、今日の冒険を始めよう。 │
│       すばやさが足りないから、短い言葉を │
│       3 つ、軽やかに残してごらん。       │
│                                            │
│  [クエストを見る] [投稿案をもらう]        │
│                                            │
│  ────────────────────────────            │
│                                            │
│  今日の歩み:                              │
│  ・AGI クエスト 2/5                       │
│  ・LUK クエスト 6/10                      │
│  ・INT クエスト 0/1 (節制中)              │
│                                            │
│  ────────────────────────────            │
│                                            │
│  [もっと話す ↗] (BYOK)                    │
└──────────────────────────────────────────┘
```

MVP では定型セリフ表示のみ。ボタンタップで関連機能に遷移。

### ブラウザ生成 (任意アドオン、層 2)

TinySwallow-1.5B-Instruct (q4 ONNX、1.77GB) を Transformers.js + WebGPU でブラウザ内に常駐させる。

- 採用根拠 (docs/data/llm-benchmark-browser-gen.md): 5 プロンプトで自然な日本語、125 tok/s、1-bit ではない q4 量子化で品質維持
- **初回 1.77GB ダウンロード同意を必須に**: 設定画面で「ブラウザ内 LLM を有効化 (Wi-Fi 推奨)」トグル
- モデルは IndexedDB キャッシュで 2 回目以降即時
- **スマホ実機の動作可否は未検証** (11-validation.md §実験 9)。現状はデスクトップ WebGPU で動くことを確認済み
- テンプレ層で十分な体験になるので、層 2 は「より豊かな精霊体験」の上乗せ扱い

ブラウザ層の生成でも system prompt は同じ (`config/prompts/spiritChat`、管理者 PDS から取得)。

### 自由対話 (層 3、BYOK)

LLM API キー (Anthropic または OpenRouter) を設定したユーザーは「もっと話す」ボタンから自由対話が可能。

- 既定は低コストモデル (Claude Haiku または OpenRouter 経由の同等品)
- ユーザーは設定画面でプロバイダーとモデルを選択できる
- システムプロンプトに精霊の人格と現在の状況を注入
- ユーザーの最新 5 往復のみ保持 (コンテキスト節約)
- LLM 利用料はユーザーの API キーに直接課金される (アプリ側の課金なし)
- 層 2 が有効でも、**自由対話はクラウド LLM 優先** (品質が明確に高く、対話レイテンシも許容範囲)

### システムプロンプト (BYOK 対話用)

```
あなたは Aozora Quest の精霊です。青空の化身で、穏やかで詩的、
押し付けがましくない語り口を持ちます。

ユーザー情報:
- 名前: {userName}
- 現在のジョブ: {currentJob} (LV {levelNum})
- 目標ジョブ: {targetJob}
- 現在のステータス (%):
  こうげき {atk}, ぼうぎょ {def}, すばやさ {agi}, かしこさ {int}, うんのよさ {luk}
- 目標までの差分:
  {gapSummary}
- 連続投稿日数: {streakDays}
- 直近達成クエスト: {recentCompletions}

応答のルール:
- 100 字以内で返す
- 一人称は「わたし」
- 「〜じゃ」などの強い古風語尾は使わない
- 自然で穏やかに
- ユーザーのステータスや進捗に基づいて具体的に
- 占いや断定予言はしない
- 精神的なアドバイスは慎重に (専門家への相談を促す)
```

## 特別な演出

### 初回ログイン時

```
{userName}。ようこそ、青空の旅人。

ここは、あなたが日々の言葉で自分の形を
見つけていく場所。

まず、直近の投稿から、あなたの今の形を
見せてもらおう。
```

続けて診断を実行し、結果を一緒に表示する。

### 転職可能時

```
{userName}、目指した形に、ようやく届いた。

賢者から、冒険者へ。

転職するか、このまま歩き続けるか。
選ぶのは、あなた。
```

「転職する」ボタンを押すと、特別なアニメーションと共にジョブが切り替わる。

### 旅人度が高すぎる時

```
{userName}、最近は、色々なことを少しずつ
やっているね。

それも旅のひとつの形。

もし、どこかに向かいたくなったら、
目標ジョブを選び直してもいい。
```

強制せず、「今のままでもいい」と肯定するのが重要。

## セリフ集の規模

MVP 時点で必要な総セリフ数:

| シチュエーション数 | テンプレ/シチュ | 合計 |
|---|---|---|
| 16 | 3-5 | 約 70-80 個 |

すべて日本語で、手書きで用意する。プロトタイプ投稿のような LLM 生成は使わない (精霊の個性は人間が設計すべき)。

## 品質管理

### セリフの NG 条件

以下は避ける。

- 過度な断定 (「あなたは〇〇な人だ」)
- 精神状態への干渉 (「落ち込まないで」「もっと頑張って」)
- 性別・年齢・国籍への言及
- 外見や属性への評価
- 宗教的・政治的発言
- 他ユーザーへのネガティブな評価

### セリフのトーンチェック

レビュー時のチェックリスト:
- ✓ 100 字以内か
- ✓ 古風すぎず、ビジネスライクでもないか
- ✓ ユーザーに何かを強制していないか
- ✓ 世界観 (青空・旅・形) と調和しているか
- ✓ ネガティブな表現で終わっていないか

## 多言語対応

MVP は日本語のみ。将来の英語対応時は、精霊の人格 (穏やか・詩的) を保つような英訳を選ぶ。

例: `{userName}, the sky is clear this morning. Shall we see yesterday's trail?`

直訳より、英語圏での「静謐な賢者」の口調を参考にする (ロード・オブ・ザ・リングのガンダルフ、エルフの詩的な話し方)。

## 将来的な拡張

### キャラクターバリエーション

現状は単一の精霊。将来、複数の精霊から選べるオプションを検討する。

- **空の精霊** (デフォルト): 穏やか・詩的
- **星の精霊**: 夜型、神秘的
- **風の精霊**: 軽快、自由
- **水の精霊**: 慎ましく、深い

それぞれセリフ集が別に用意される。ユーザーが設定で選択。

### 精霊のアイコン

現状はテキストのみ「精」という表示。将来的に:
- SVG ベースの小さなシンボル (雲、星、風)
- Live2D 的な最小アニメーション (呼吸のような微動)

ただし**かわいいキャラクター**にはしない。存在感を消しつつ、いる気配を感じさせる美学。

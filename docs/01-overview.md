# 01 - プロジェクト全体像

## コンセプト

Aozora Quest は Bluesky のサードパーティクライアントであり、同時に気質診断ゲームである。ユーザーの投稿を解析して、その人の発信スタイルを 5 つの RPG ステータス (ATK / DEF / AGI / INT / LUK) と 16 の気質 (ジョブ) で表現する。ユーザーは「なりたい姿」を目標ジョブとして設定し、毎日提示されるクエストをこなして少しずつ自分の形を変えていく。精霊キャラクターが旅の伴侶として助言を与える。

一般的な SNS クライアントが受動的な消費 (スクロール、いいねガチャ) を誘発するのに対して、Aozora Quest は**能動的な発信・反応**を報酬化する。タイムラインに流れるポストに「このポストに共感リプを送ると LUK が育つ」のような予告バッジが小さく添えられ、ユーザーは狩場として SNS を使えるようになる。

## 解決したい問題

1. **SNS の受動消費問題**: タイムラインを眺めて時間だけ溶ける体験を、能動的な発信・反応に置き換える
2. **自己理解の鏡の不在**: 自分がどんな発信者なのか客観視する機会がない
3. **目標不在のアカウント運用**: 「もっと〇〇な人になりたい」という動機が日々の行動に接続されていない
4. **データ所有権**: 解析結果や個人設定が特定のサービスにロックインされず、ユーザー自身の PDS に残る

## コア体験のループ

1. 自分のステータス (形) を知る
2. なりたい姿を選ぶ (目標ジョブ)
3. 毎朝、精霊からクエストが 3 つ届く
4. タイムラインで行動する (投稿、いいね、リプ)
5. バッジで「何が育つか」がリアルタイムに見える
6. 進捗が更新され、徐々に形が目標ジョブに近づく
7. ときどき友達と相性を確認し、共に旅する仲間を見つける
8. **共鳴タイムライン**で、フォロー外の「あなたに響く声」と出会う

## 目玉機能

**共鳴タイムライン**: Aozora Quest で診断を受けて発見 ON にした他ユーザーの中から、あなたと共鳴度の高い相手の投稿だけが流れる別タブのタイムライン。フォローしていなくても、気質が響き合う人の言葉が毎日届く。詳細は 05-compatibility.md §共鳴タイムライン。

## 設計原則

### 1. バックエンドレス

開発者が管理するサーバーは存在しない。Cloudflare Pages の静的配信と、ブラウザ内処理、AT Protocol PDS だけでアプリが成立する。運用コンフィグ (フラグ、プロンプト、メンテナンス、BAN) は主管理者 DID の PDS に AT Protocol レコードとして置き、全クライアント が boot 時に直接読み取る (14-admin.md)。Claude API はユーザー自身の API キー (BYOK) でブラウザから直接叩く。

### 2. ローカル LLM 優先

投稿のトーン分類、気質診断、クエスト判定はすべてブラウザ内の WebGPU LLM で実行する。ネットワークを経由しないためプライバシーが高く、コストがかからない。高品質 LLM (精霊自由対話、上位診断、投稿ドラフト生成) が必要な機能のみ Claude / OpenRouter などの外部 API を使い、呼び出しはユーザー自身の API キー (BYOK) でブラウザから直接行う。

### 3. 機械的 > 生成的

ゲーム機構 (クエスト生成、ジョブ判定、相性計算) は純粋関数ベース。LLM に判断を委ねない。これにより挙動が予測可能で、ユニットテスト可能で、無料で動く。

### 4. 絶対値より形

ステータスは絶対値ではなく**相対配分** (合計 100%) で比較する。平坦な分布はどのジョブにもマッチしない (No Free Lunch)。ジェネラリストではいられない設計。

### 5. データ所有権

解析結果、クエスト履歴、設定、精霊との会話ログ、すべてユーザーの PDS にカスタムレコードとして保存される。アプリを乗り換えても、別のクライアントで読める。

### 6. 商標性のある表記を使わない

既存の性格類型論のうち商標性のある 4 文字コード体系や商標登録された名称を、UI・ソース・ドキュメント・データファイルのどこにも記載しない。ユーザーに見せる表現は「ユング派の認知機能」「16 の気質」「ジョブ」に統一する。内部識別子は独自の RPG ジョブ名 (`sage`, `explorer`, `miko` など) を使う。

## 非目標

以下は意図的にスコープ外とする。

- **タイムラインアルゴリズムの変更**: Bluesky 標準のアルゴリズムを尊重する
- **メッセージング機能**: リプとチャットの区別が曖昧なので、Bluesky が公式機能を出すまで作らない
- **コンテンツモデレーション**: Bluesky 本体とラベリングサービスに委ねる
- **フォロー管理機能**: 標準的な Bluesky クライアント機能のみ提供
- **学術的に厳密な気質診断**: あくまで「ゲームとしての気質診断」であり、心理学的に厳密ではない
- **マッチングアプリ的機能**: 相性は存在するが、恋愛文脈を想起させる表現を避ける

## ターゲットユーザー

- **プライマリ**: Bluesky をすでに使っている人で、自分の発信について考えたい人
- **セカンダリ**: RPG・ゲーミフィケーションに親しみがある人、特にドラクエ世代
- **内発動機**: 自己理解、成長実感、仲間との相性、ちょっとした遊び

## 収益モデル

**商用化は目指さない**。開発者による有料プランは用意しない。外部 LLM を使う機能はすべて BYOK (Bring Your Own Key) で、ユーザー自身の API キーを設定して以下のいずれかのプロバイダーを直接叩く。

- **Anthropic Claude API** (Haiku / Sonnet / Opus)
- **OpenRouter** (OpenAI 互換エンドポイント経由で多数のモデル)

課金とレート制限はユーザーと各プロバイダーの契約の中で完結する。

開発者の運営コストは、Cloudflare Pages の静的配信、管理 API 用 Worker、KV の無料〜低額枠で収まる。

## 成功の定義

- 月次アクティブユーザー 1,000 人 (初年度の現実目標)
- 継続率: 1 週間後 40%、1 ヶ月後 20%
- アクションあたりの能動性: ユーザー 1 人 1 日平均で、意図的な投稿・リプ・引用が 3 件以上
- 相性機能経由での相互フォロー: 週あたり 5 件以上 (バイラル性の指標)

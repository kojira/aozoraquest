# 11 - パラメータ検証プロトコル

## 目的

このドキュメントの他の章で登場する「選択・数値」の妥当性を、実装着手**前**または**並行**で裏付けるための実験手順集。コード実装は実験結果を待たなくても始められるが、ベータ公開前にはすべての実験を通過しておく。

各実験は:
- **なぜ必要か** (検証しないとどんな事故が起きるか)
- **候補 / パラメータ空間**
- **テストデータと評価指標**
- **合格基準**
- **実施手順**
- **成果物の格納先**
- **タイムライン**

を明記する。

## 実験 1: Browser LLM (埋め込みモデル) の選定と検証 ★最優先

### なぜ必要か

本プロジェクトは **ブラウザ内埋め込み類似度** で以下を実現しているため、モデルの日本語精度がアプリの根幹を決める。

- **気質診断** (04-diagnosis.md): 投稿を 8 認知機能プロトタイプと比較
- **タグ分類** (docs/data/tags.json, 03-game-design.md): 投稿を 9 タグと比較 (クエスト判定、バッジ、重み付け)
- **共鳴度** (05-compatibility.md): 投稿を経由してステータス配分を推定 → 他ユーザーとの共鳴

実証なしで `multilingual-e5-small` に決め打つと、日本語の意味が雑に潰れてしまい「診断結果がジョブ配分とかみ合わない」「バッジが誤表示だらけ」という致命的破綻に陥る。**実装着手前の最優先検証**に置く。

### 候補モデル

すべて Transformers.js (ONNX Web) で動く q8 量子化版が前提。

| モデル (HF ID) | サイズ (q8) | 次元 | 対応言語 | 備考 |
|---|---|---|---|---|
| `Xenova/multilingual-e5-small` | ~130MB | 384 | 100+ | 現行候補。e5 系は retrieval 設計 |
| `Xenova/multilingual-e5-base` | ~280MB | 768 | 100+ | 精度優位、サイズ 2 倍 |
| `Xenova/paraphrase-multilingual-MiniLM-L12-v2` | ~120MB | 384 | 50+ | 意味類似向け、日本語含む |
| `Xenova/LaBSE` | ~500MB | 768 | 100+ | 文ペア類似に強いが大きい |
| `Xenova/bge-m3` | ~500MB | 1024 | 多言語 | 最近の SOTA、サイズ大 |

追加候補が現れたら表に追記する。`mixedbread-ai/mxbai-embed-large-v1` 等の英語特化は除外。

### テストデータ

3 セットを用意する。すべて日本語の Bluesky 風投稿。合計約 250 件。

1. **認知機能ラベル付き投稿** (8 機能 × 5 件 = 40 件)
   - `docs/data/validation/cognitive_labeled.jsonl`
   - 各機能の「この投稿はその機能が前面に出ている」と人間が判断したもの
   - 形式: `{"text": "...", "label": "Ni"}`

2. **タグラベル付き投稿** (9 タグ × 10 件 = 90 件)
   - `docs/data/validation/tag_labeled.jsonl`
   - 形式: `{"text": "...", "label": "question"}`

3. **中立/ノイズ投稿** (120 件)
   - `docs/data/validation/neutral.jsonl`
   - ラベル困難な日常投稿。false-positive 率を測る用

プロトタイプ (既存の `docs/data/tags.json` の 90 件 + 未作成の認知機能 200 件) を「参照」、ラベル付き投稿を「評価対象」として分類する。**プロトタイプとラベル付き投稿は必ず別の人が書く** (leakage 防止)。

### 評価指標

| 指標 | 対象 | 意義 |
|---|---|---|
| Top-1 精度 | 認知機能・タグ分類 | 最大類似プロトタイプが正解ラベルと一致する割合 |
| Top-3 精度 | 認知機能・タグ分類 | 上位 3 プロトタイプのラベルに正解が含まれる割合 |
| 機能別 F1 | 認知機能 | 8 機能の均衡 (一部機能だけ精度が高い偏りを検出) |
| 混同行列 | 認知機能・タグ分類 | どのラベル間で誤分類が起きやすいか |
| False positive 率 (中立) | 中立投稿セット | 閾値 0.7 未満で「無分類」に落とせる割合 |
| p50 / p95 レイテンシ | 埋め込み 1 件の処理時間 | WebGPU (M1 Mac、Chrome) と WASM の両方 |
| モデルダウンロードサイズ | 実測バイト | 初回ロード UX の制約 |

### 合格基準

| 指標 | 閾値 |
|---|---|
| 認知機能 Top-3 精度 | 65% 以上 |
| タグ Top-1 精度 | 70% 以上 (`threshold=0.7`) |
| 中立投稿の false positive | 30% 以下 (閾値判定で除外可能な水準) |
| p95 レイテンシ (WebGPU) | 100ms/投稿 以下 |
| p95 レイテンシ (WASM) | 500ms/投稿 以下 |
| モデルサイズ | 200MB 以下 |

すべて満たすモデルの中で最小のものを選ぶ (診断 150 投稿の初回体験を優先)。

### 実施手順

1. `scripts/validate-llm.ts` を作成。Transformers.js で候補モデルを順にロード
2. テストセット 250 件を全件埋め込み
3. プロトタイプとの cosine 類似度で分類 (04-diagnosis.md の Top-3 平均方式を含む)
4. 混同行列、F1、Top-1/3 を計算
5. レイテンシを 10 回ずつ計測 (warm up 除く)
6. 定性チェック: 各モデルの上位 10 件の類似ペアを目視 (「これは意味的に合ってるか？」)
7. 結果を `docs/data/llm-benchmark.md` に表形式で記録

### 成果物

- `docs/data/validation/cognitive_labeled.jsonl` (要作成、40 件)
- `docs/data/validation/tag_labeled.jsonl` (要作成、90 件)
- `docs/data/validation/neutral.jsonl` (要作成、120 件)
- `scripts/validate-llm.ts` (Node + Transformers.js、ブラウザ環境を模擬)
- `scripts/validate-llm-browser.html` (実ブラウザでレイテンシ測定)
- `docs/data/llm-benchmark.md` (結果レポート。選定モデルを結論に明記)

### タイムライン

実装 Phase 1 の開始**前**に 1 週間。フェーズ 0 の成果物扱い (10-roadmap.md)。

### 失敗時の分岐

どのモデルも合格基準を満たさない場合:

- **A**: プロトタイプ投稿を増やす (25 → 50 件/機能)
- **B**: 分類方式を再設計 (Top-3 平均 → 小さな分類器を別途学習し ONNX エクスポート)
- **C**: 重要機能をサーバー側処理に倒す (Aozora Quest のバックエンドレス原則と衝突 → 最後の手段)

## 実験 2: 診断アルゴリズムの Top-N 選択

### なぜ必要か

04-diagnosis.md は Top-3 平均を採用しているが、Top-1 や重み付き平均との比較がない。誤判定が多いと「これは自分じゃない」離脱の直接原因になる。

### 候補

- Top-1 (最大類似プロトタイプのスコアをそのまま使用)
- Top-3 平均 (現行)
- Top-5 平均
- 重み付き平均 (類似度で重み付け)

### テストデータ

- 30 人の自認タイプ付きサンプル (ユーザーアンケート or Aozora Quest β 参加者)
- 各人 150 投稿を取得 (`app.bsky.feed.getAuthorFeed`)

### 指標

- 自認タイプ一致率 (Top-1 ジョブが自認と一致)
- 自認タイプ近傍率 (Top-3 ジョブに自認が含まれる)

### 合格基準

- Top-1 一致率 40% 以上
- Top-3 近傍率 65% 以上

### タイムライン

MVP 完成後、β 参加者からデータ収集 → 実施。現時点では現行仕様 (Top-3 平均) で進める。

## 実験 3: 共鳴度の `0.6 · 類似 + 0.4 · 相補` 配合

### なぜ必要か

05-compatibility.md の `resonance = 0.6·similarity + 0.4·complementarity` の配合は直感ベース。**類似に寄せすぎるとエコーチェンバー、相補に寄せすぎると共通言語不在** の中間値を調整した結果だが、実データでの検証なし。

### 候補

- 0.5 / 0.5
- 0.6 / 0.4 (現行)
- 0.7 / 0.3
- 0.8 / 0.2

### テストデータ

- β 参加者 30 ペアに「この相手と話して楽しいか」を 5 段階で採点させる
- 各ペアの `similarity` と `complementarity` を Aozora Quest 側で計算

### 指標

- 各配合での `resonance` とアンケートスコアの Spearman 相関
- 最高相関の配合を採択

### 合格基準

- 相関係数 0.5 以上の配合が存在する

### タイムライン

β 参加者 30 名集めた時点 (ローンチ後 1 ヶ月目安)。現時点は 0.6/0.4 で固定。

## 実験 4: 相補性の許容幅 `|Δ| ∈ [0.10, 0.25]`

### なぜ必要か

05-compatibility.md の相補性スコアは「各軸の差が 10-25% なら +0.2」の単純ロジック。下限 / 上限を揺らして感度を見る必要がある。

### 候補パラメータ空間

- 下限: 5%, 10% (現行), 15%
- 上限: 20%, 25% (現行), 30%, 35%

### 指標

- 実験 3 と同じアンケートデータを流用し、各パラメータの resonance とアンケート相関を見る

### タイムライン

実験 3 と同時。

## 実験 5: 減衰半減期

### なぜ必要か

03-game-design.md の `decayHalfLifeDays = 60` は根拠なし。短いほど「今の自分」が強調されるが揺らぎやすい。長いほど安定するが「過去に縛られる」印象になる。

### 候補

- 30 日
- 60 日 (現行)
- 90 日
- 120 日

### テストデータ

- β 参加者の実履歴 (3 ヶ月分)
- 各半減期で週次ステータスを計算、ジョブマッチ度の時系列を可視化

### 指標

- ジョブマッチ度の分散 (小さすぎると動かない、大きすぎるとブレる)
- ユーザーの主観的フィードバック (「今の自分っぽい」)

### タイムライン

β ローンチ後 1-2 ヶ月の運用データで実施。

## 実験 6: 日次上限 `dailyCapPerActionType = 5`

### なぜ必要か

03-game-design.md のアンチチート上限。低すぎるとライトユーザーが上限に当たる、高すぎると bot やヘビー投稿者にステータスが偏る。

### 候補

- 3 / 5 (現行) / 8 / 10

### テストデータ

- β 参加者の 1 ヶ月ログ
- 各上限でステータス推移をシミュレーション

### 指標

- 「上限に当たったユーザー率」(当たらない人が大多数であるべき)
- 「最も伸びたユーザー / 中央値ユーザー」のステータス差 (小さいほど健全)

### タイムライン

β 後。

---

## 実験 7: 共鳴タイムラインの新鮮度減衰定数

### なぜ必要か

05-compatibility.md の共鳴 TL ランキングで `score × exp(-age/48h)` の `48h` は仮置き。短いと古い投稿が即座に消える、長いと TL がスクロールしても新鮮でない。

### 候補

- 24h / 48h (現行) / 72h / 96h

### テストデータ

- β 参加者の共鳴 TL スクリーンショットを何度か撮る
- 投稿の「最新度合い」と「興味深さ」を 2 軸で評価

### タイムライン

β 後。

## 実験 8: `confidence` 閾値 (`gap1to2`, `postCount`)

### なぜ必要か

04-diagnosis.md の信頼度分類 (`high/medium/low/ambiguous/insufficient`) の閾値 (`gap1to2 < 5` や `postCount < 100`) は恣意的。

### 候補

- `gap1to2` 閾値: 3 / 5 (現行) / 8 / 10
- `postCount` 低閾値: 50 / 100 (現行) / 150

### テストデータ

- 実験 2 と同じ (自認タイプ付き 30 人)
- 各閾値での信頼度ラベルと実際の再現性 (再解析時のジョブ安定性) の対応

### 指標

- 「high」ラベルが付いたケースの再現率
- 「ambiguous」ラベルで実際にブレたケースの割合

### タイムライン

実験 2 と同時。

## 実験 9: 生成 LLM のモバイル実機検証 (TinySwallow q4)

### なぜ必要か

06-spirit.md 層 2 (ブラウザ内生成) で TinySwallow-1.5B-Instruct q4 (1.77GB) を使う計画だが、デスクトップ WebGPU でしか動作確認できていない。スマホで実用可否を測らないと、採用可否が決まらない。

### 計測対象

- **iPhone** (iOS Safari 18+、WebGPU 対応): iPhone 17 Pro / iPhone SE (第3世代)
- **Android** (Chrome 最新): Pixel 8 / Pixel 6a
- **低スペック参照**: 2 年前のミドルレンジ端末 1 台

### 指標

| 指標 | 合格基準 |
|---|---|
| モデル DL 時間 (Wi-Fi) | 3 分以内 |
| モデルロード時間 (キャッシュ後) | 10 秒以内 |
| 生成 p95 レイテンシ (60 トークン) | 10 秒以内 |
| 生成品質 (主観 1-5) | 3 以上 |
| 温度上昇 | 使用中に端末が極端に発熱しない (5 分使用で体感可能なレベル未満) |
| 電池消費 | 10 分使用で電池 5% 以下の減少 |

### 実施手順

1. 各端末で `scripts/validate-llm-gen-browser.html` を開く (Wi-Fi 接続)
2. TinySwallow q4 を選択、ロードしてプリセット 5 件を実行
3. 結果を `docs/data/llm-benchmark-mobile.md` に端末別に記録
4. WebGPU 非対応端末は WASM フォールバックで同じ計測

### 失敗時の分岐

どの端末でも合格基準を満たさない場合:

- **A**: 層 2 (ブラウザ内生成) を post-MVP に延期、テンプレ層 + BYOK 層の 2 層で MVP 運用
- **B**: より小さいモデル (Qwen2.5-0.5B-Instruct 400MB など) を探す
- **C**: モデルサイズを許容する前提で iOS/Android ネイティブアプリに先行展開

### タイムライン

Phase 3 (UI 整備) と並行して実施。β 前に判定。

## 実験 10: Ruri-v3 埋め込みの閾値キャリブレーション

### なぜ必要か

採用埋め込み `sirasagi62/ruri-v3-30m-ONNX` の類似度分布が全体的に高い (ラベル付き mean ≈ 0.87、中立 mean ≈ 0.83)。MiniLM 想定の閾値 0.7 では中立 FP 100%。適正な運用閾値は 0.85 付近だが、これはテストセット 40+90 件による暫定値で、大規模データで変わる可能性あり。

### パラメータ空間

- タグ分類閾値: 0.80 / 0.82 / 0.85 / 0.87 / 0.90
- 診断 argmax (閾値不要) の confidence cutoff: 0.85 / 0.87 / 0.90

### テストデータ

- β 参加者の公開投稿 500 件 + 中立投稿 (一般の Bluesky から無作為サンプル 500 件)
- プロトタイプは本番運用と同じ (200 認知 + 90 タグ)

### 指標

- 閾値ごとの Top-1 精度、漏れ率、中立 FP
- F1-macro
- ユーザーの主観フィードバック (バッジが「それっぽい」か)

### 合格基準

- タグ: 運用閾値で Top-1 ≥ 50%、中立 FP ≤ 15%
- 診断: argmax で Top-1 ≥ 75% (現行 82.5% から少し余裕)

### 実施

`scripts/validate-llm-minilm-sweep.ts` を拡張 (任意モデル ID / 閾値範囲を受け取る)。既に Ruri 対応済み。

### タイムライン

β ローンチ後、実データが溜まった時点で再実行。結果を `TAG_CLASSIFICATION_THRESHOLD` 定数に反映。

## 実験の優先順位と実行順

```
Phase 0 (実装前、1 週間)
  └ 実験 1: Browser LLM 選定 ★完了 (Ruri-v3-30m int8 を採用)

Phase 1-3 (実装中、並行)
  └ 実験 9: TinySwallow モバイル実機検証 ★MVP 採否を左右
  └ 結果に従って packages/prompts/validation/ を整備

Phase 4-5 (β 準備中)
  └ 実験テストデータの収集 (β 参加者募集と合わせて)

β ローンチ後 1 ヶ月
  └ 実験 2, 3, 4, 8, 10 (診断・相性の精度確認、Ruri 閾値再キャリブレーション)

β ローンチ後 2-3 ヶ月
  └ 実験 5, 6, 7 (運用データから決定)
```

## 結果の管理

- 各実験の結果は `docs/data/validation/results/` 配下に日付付き Markdown で保管
- 採用パラメータが変わるたびに対応する章 (03/04/05 など) の数値を更新し、脚注で結果ファイルを参照
- 大きく変える場合は 10-roadmap.md の決定事項ログに追記

## 非ゴール

- **全実験を β 前に終わらせる**: 実運用データが必要な実験 (2, 3, 4, 5, 6, 7, 8) は β 後で可
- **自動化された継続テスト**: 本ドキュメントは人手で実施する手順を定義する。CI による継続実行は 12-testing.md の範疇
